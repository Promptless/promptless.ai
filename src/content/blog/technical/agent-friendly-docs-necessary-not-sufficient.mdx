Yesterday we became the first documentation site to reach a perfect score of 100 on the Agent-Friendly Docs benchmark.

We're proud of this. Our team put real work into making our docs accessible to AI agents: static Markdown exports for every page, a comprehensive `llms.txt` index, content-negotiation middleware so agents can request `text/markdown` directly, and hidden directives on every page pointing to our documentation map. When Claude Code, Cursor, or any other coding assistant fetches our docs, it gets clean, structured content instead of HTML soup filled with navigation chrome.

But here's the thing: a perfect accessibility score is table stakes. Necessary, but nowhere near sufficient. And if you're optimizing for agent accessibility without thinking about what comes next, you might be making things worse.

## The accessibility problem is mostly solved

The basic mechanics of making docs agent-readable are well understood now. Serve Markdown or clean HTML. Maintain an `llms.txt` file that acts as a site map for AI. Strip out JavaScript-heavy components that don't survive fetch. Use semantic headings. These are the checkboxes.

Most documentation platforms handle this automatically. Mintlify, GitBook, ReadMe, and others have shipped agent-friendly features as defaults. If your docs are on a modern platform, you probably score reasonably well on accessibility metrics without doing anything special.

The harder problem is what happens after the agent successfully reads your docs.

## Agents are more credulous than humans

Here's something that doesn't get discussed enough: AI agents believe what they read in a way that humans don't.

When a human developer reads documentation, they bring healthy skepticism. They test code examples before trusting them. They notice when something feels outdated. They cross-reference against the actual API behavior. If the docs say one thing and the code does another, they believe the code.

Agents don't do this. They take your docs at face value. If your authentication guide says to use an API key in a header called `X-Auth-Token` but you deprecated that six months ago in favor of Bearer tokens, the agent will confidently generate code using the deprecated pattern. It won't notice the mismatch. It won't think to test it. It will do exactly what your docs told it to do.

This is a feature of how language models work. They're trained to be helpful, to complete tasks based on the information they're given. They don't have the adversarial instinct that humans develop from years of being burned by bad documentation. They trust you.

Which means they're easier to mislead.

## The blast radius of bad docs just increased

Every inaccuracy in your documentation used to affect one developer at a time. Someone would follow your quickstart, hit an error, maybe file a support ticket or give up and choose a competitor. Painful, but contained.

Now multiply that by every AI coding assistant your potential users have installed. A single outdated code example doesn't just mislead the developer who reads it—it gets served to Claude, to Copilot, to Cursor, to Windsurf, to hundreds of coding agents that might recommend your product to their users. Each one ingests your docs, believes them completely, and generates code that doesn't work.

The debugging experience is worse too. When a human reads bad docs, they at least know which page they read and can report the problem. When agent-generated code fails, the developer often has no idea where the agent got its information. They don't know which of your docs pages was the source. They just know your product doesn't work the way the AI said it would.

You've made it easier for agents to access your docs. You've also made it easier for your docs' inaccuracies to propagate at scale.

## Accuracy is the new competitive advantage

The logical conclusion: if your docs are going to be read primarily by AI agents, accuracy and clarity matter more than they ever did for human readers.

This isn't just about catching typos. It's about:

**Stating things exactly.** Don't say "you might need to configure authentication" when you mean "you must configure authentication before making any API calls." Agents don't handle ambiguity well. They'll pick whichever interpretation seems more plausible, which may not be correct.

**Keeping examples current.** Code examples are the highest-signal content for agents. When an LLM needs to generate code using your API, it will lean heavily on your examples. If those examples use deprecated methods, deprecated syntax, or deprecated patterns, the agent will reproduce them.

**Versioning carefully.** If your product has multiple versions with different behaviors, make sure the docs clearly indicate which version they apply to. An agent that retrieves your v2 docs and generates v3 code will create a confusing mess for the developer who has to debug it.

**Documenting failure modes.** Agents struggle with unwritten knowledge. The constraints and edge cases that experienced users know intuitively—"oh, you can't do X if Y is already configured"—need to be explicit. If they're not in the docs, the agent won't know about them.

**Writing clearly.** Dense, jargon-heavy prose that a motivated human can puzzle through will trip up an agent. Short, declarative sentences. One concept per paragraph. Fewer clever phrasings, more direct statements.

## Why we built Promptless

This is why we started Promptless. Not because we were excited about `llms.txt` or Markdown exports—those are solved problems. We started Promptless because we saw that documentation accuracy was about to become a much harder and much more important problem.

When your docs are primarily consumed by AI agents, you need a system that catches every inaccuracy before it propagates. You need docs that update automatically when your product changes. You need verification that what you've written actually matches what your API does.

Scoring 100 on agent accessibility was a milestone. Making sure every page deserves that accessibility score is the real work.
