This example teaches Promptless to automatically review a documentation PR opened by someone *other than* Promptless, on two dimensions: (a) documentation quality and style, and (b) correctness against the source code. It is a worked instance of the general process described in [Teach Promptless a custom task](/docs/get-the-most-out/teach-promptless-a-custom-task) — a written review workflow paired with a `github_pr` trigger. The steps below first write the workflow, then configure the trigger, then save both. They describe the intended end-to-end design; see [What's available today](#what-promptless-does-on-each-qualifying-pr) for what Promptless actually delivers right now.

## Prerequisites

Before you set up this task, make sure you have:

- **The GitHub integration connected** to your documentation repository — for example, `acme/docs`.
- **A Knowledge Base with `PROMPTLESS.md`**, where the review workflow will live.
- **Org-admin access** to add the review workflow to `PROMPTLESS.md` and the `review-inbound-doc-prs` `github_pr` trigger to `promptless.yaml` — or the ability to ask Promptless to make those edits for you.
- **Open-source or public-repo setups** can follow the [Open Source Quickstart](/docs/start-here/open-source-quickstart) to connect the repo, since this task reviews outside-contributor PRs.

Set up the task in three steps: write the workflow, configure the trigger, then save both. The caution notes below flag which parts describe the intended design versus what runs today.

<Steps>
  1. **Write the workflow.** The workflow captures *how* to review a contributor documentation PR: when it applies, what to check, and how to act. Paste an instruction block like this into `PROMPTLESS.md`. Because this block lives inline in `PROMPTLESS.md`, its opening "When ...:" condition line together with the instructions beneath it serves as the required activation hook — the "If, then," where the instructions are the "then" — so it fires reliably, because `PROMPTLESS.md` is always loaded into the agent's instructions. (If you instead keep the workflow in a Skill, you additionally need a hook in `PROMPTLESS.md` that names the skill and its doc collection — covered in the note after the block.) See the hook requirement in [Teach Promptless a custom task](/docs/get-the-most-out/teach-promptless-a-custom-task).

     ```markdown
     # Inbound documentation PR review

     When a pull request is opened in the docs repo (acme/docs) by an author who is
     not Promptless, and the PR touches documentation content:

     - No-op on Promptless-authored PRs. There is no per-author config filter, so
       check the PR author and stop if it is the Promptless bot.
     - Review the changes on two dimensions:
       1. Quality and style — check that the edits match your voice, structure, and
          formatting conventions, plus any Vale rules the repo enforces.
       2. Correctness against the source code — verify the PR's claims against the
          actual product source, and flag or fix anything stale or unsupported.
     - Lean toward direct edits on the PR branch for clear fixes; comment only for
       ambiguity or judgment calls that need the author to resolve them.
     - Always post a top-level summary comment describing what you reviewed, what you
       changed, and any open questions. This is required even when nothing changed.
     - Set a GitHub review state by default (approve or request changes),
       configurable.
     ```

     The "direct edits" and "review state" instructions describe the intended design; see [What's available today](#what-promptless-does-on-each-qualifying-pr) for what runs right now. Paste the block as written — Promptless acts on the review and summary-comment instructions today, and it will pick up the direct-edit and review-state instructions automatically once that capability ships. (If you defined this review workflow instead as an Agent Skill living in the doc collection — for example, under `.claude/skills/` in `acme/docs` — the `PROMPTLESS.md` hook must name that doc collection and skill, e.g., "When a contributor PR is opened in `acme/docs`, use the `contributor-workflow` skill from the Acme doc collection.")

  2. **Configure the trigger.** Scope a `github_pr` trigger to your documentation repository so the workflow fires when a contributor opens a PR there. This trigger listens on the docs repo and narrows to your documentation content:

     ```yaml
     triggers:
       review-inbound-doc-prs:
         trigger_type: github_pr
         match:
           - repos:
               - acme/docs
             trigger_on:
               - opened
               - first_approval
               - merge
             trigger_directories:
               - docs/
     ```

     - **`opened`** starts the review as soon as a contributor opens the PR — the primary moment for this task.
     - **`first_approval`** lets Promptless weigh in around the first human approval.
     - **`merge`** covers the point where the PR lands.
     - **`trigger_directories: docs/`** narrows the trigger to your documentation content, so PRs that touch only unrelated files do not fire it.

     The `github_pr` trigger supports exactly these three events. For the full set of options, see [GitHub PRs](/docs/connect/triggers/github-prs) and the [Configuration Reference](/docs/reference/configuration-reference#triggers). Organization admins edit and save `promptless.yaml`, or you can ask Promptless to make the change for you.

     <Aside type="caution">
       The `github_pr` events are `opened`, `first_approval`, and `merge` only. There is no "updated"/synchronize/re-review-on-push event today, so you can't configure automatic re-review when a contributor pushes new commits. To re-review after new commits, request it manually — an `@promptless` mention in a PR comment re-triggers a review (see [GitHub PRs](/docs/connect/triggers/github-prs)).
     </Aside>

  3. **Save the PROMPTLESS.md updates.** Commit the workflow block into `PROMPTLESS.md`, and commit the trigger into `promptless.yaml`. There are two ways to do this:

     - An organization admin edits the files directly in the [Agent Knowledge Base](/docs/get-the-most-out/agent-knowledge-base) (in the left sidebar, open **Configure** and select **Agent Knowledge Base**).
     - You can ask Promptless to make these changes for you — it edits `PROMPTLESS.md` and `promptless.yaml` and commits the change.
</Steps>

## What Promptless does on each qualifying PR

When a contributor opens a qualifying PR, Promptless:

1. Detects the qualifying PR.
2. Reads the full PR context — title and description, the diff, review comments, and commits.
3. Reviews the changes on both dimensions: quality and style, and correctness against the source code.
4. Posts a top-level summary comment describing what it reviewed and what it found.

<Aside type="caution">
  Today the review agent posts a **top-level summary PR comment**, which is available now. It cannot yet push edits directly to a contributor's PR branch, submit a formal GitHub review state (approve or request changes), or leave inline review comments — the team is actively working on the permissions for this. So the "direct edits" and "review state" parts of this example describe the intended design; today's guaranteed deliverable is the top-level summary comment. In practice, today this delivers an automatic first-pass review comment on every qualifying PR, but it does not yet act as an enforceable branch-protection or required-status-check gate.
</Aside>
