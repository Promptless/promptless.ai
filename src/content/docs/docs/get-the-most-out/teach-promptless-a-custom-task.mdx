Promptless ships with default behaviors, like reviewing code PRs for documentation impact, but you can teach it new or customized tasks on top of those defaults. A custom task pairs two ingredients: a **trigger** that decides when Promptless runs, and a **workflow written as instructions** that describes what it does. This page covers the general process, including prerequisites, the workflow, the trigger, and how a run unfolds. A complete example lives on its own page: [Example: review inbound documentation PRs](/docs/get-the-most-out/teach-promptless-a-custom-task/review-inbound-doc-prs).

## The two ingredients of a custom task

Every custom task combines two pieces:

- **A trigger** determines when Promptless runs — you configure it in the `triggers` section of the [Configuration page](https://app.gopromptless.ai/configuration).
- **A written workflow** is the set of instructions the agent follows when that trigger fires — it lives in `PROMPTLESS.md` in your Knowledge Base and/or as a custom Agent Skill in your docs repo.

The agent follows written instructions well, so the guidance you provide can redirect or augment its built-in default behavior.

This trigger-plus-instructions pattern is general: you pair written instructions with [any supported trigger type](/docs/reference/configuration-reference#trigger-types), not just `github_pr`, so the same mechanism drives many different custom tasks. The worked example at the end of this page shows one end-to-end illustration.

<Aside type="note">
  This page builds on [How Promptless learns your docs](/docs/connect/doc-locations/how-promptless-learns-your-docs), which covers where Promptless reads custom guidance — `AGENTS.md`/`CLAUDE.md` in your docs repo, Agent Skills under `.claude/skills/`, `.agents/skills/`, or `.cursor/skills/`, and `PROMPTLESS.md`.
</Aside>

## Prerequisites

Before you build a custom task, make sure you have:

- **The relevant integration or trigger source connected** — for example, the GitHub integration for a `github_pr` task.
- **A Knowledge Base** where your `PROMPTLESS.md` and/or a custom Agent Skill live.
- **Org-admin access** to edit `PROMPTLESS.md` and `promptless.yaml` — or the ability to ask Promptless to make those edits for you.

Concrete, per-task prerequisites live on each worked example's page.

## Write the workflow

The workflow is the *what*: the concrete instructions the agent follows once a trigger fires. Instructions live in `PROMPTLESS.md`, and/or as a custom Agent Skill in your docs repo. To learn what `PROMPTLESS.md` is and where skills live, see [How Promptless learns your docs](/docs/connect/doc-locations/how-promptless-learns-your-docs); to view and edit Knowledge Base files, open the [Agent Knowledge Base](/docs/get-the-most-out/agent-knowledge-base).

There are two ways to update these instructions:

1. An organization admin edits `PROMPTLESS.md` in the Agent Knowledge Base (in the left sidebar, open **Configure** and select **Agent Knowledge Base**).
2. You can ask Promptless to make these changes for you — for example, by requesting the update the next time you're working with Promptless — and it edits `PROMPTLESS.md` (or the custom skill) and commits the change.

Good workflow instructions are concrete. Spell out what to check, what to prioritize, and what output to produce, so the agent knows exactly what a successful run looks like.

These custom workflows are effectively Agent Skills, and you can format them that way. Keep the workflow either in `PROMPTLESS.md` in your Knowledge Base, or as an Agent Skill in a doc collection (your docs repo, under `.claude/skills/`, `.agents/skills/`, or `.cursor/skills/`). One rule matters here: if the workflow is defined as an Agent Skill that lives in a doc collection, the `PROMPTLESS.md` hook (see the requirement below) must name *which* doc collection the skill lives in — for example, "When a contributor PR is opened in `acme/docs`, use the `contributor-workflow` skill from the Acme doc collection."

<Aside type="caution">
  For a custom workflow to activate reliably, you *must* have an explicit "If, then" hook in `PROMPTLESS.md` — of the form "If \[condition], then \[do this]." This is a requirement, not a suggestion. If the workflow is written inline in `PROMPTLESS.md`, the condition line together with the instructions beneath it is that hook — the instructions are the "then," so there is nothing separate to name. If the workflow instead lives in an Agent Skill that loads on demand, the `PROMPTLESS.md` hook must name the specific skill/workflow so it points to it. `PROMPTLESS.md` is loaded into the agent's instructions on every run, so a hook there is always in front of the agent, whereas Agent Skills are only loaded on demand — when a task matches a skill's description. A workflow that lives only in a Skill may not fire consistently unless a hook in `PROMPTLESS.md` points to it. See [How Promptless learns your docs](/docs/connect/doc-locations/how-promptless-learns-your-docs) for how these files are loaded.
</Aside>

## Configure the trigger

The trigger is the *when*. You manage triggers on the [Configuration page](https://app.gopromptless.ai/configuration) in the `triggers` section, which is backed by a single `promptless.yaml` file in your Agent Knowledge Base, and saving commits the change there. You can also ask Promptless to make these trigger changes for you — Promptless edits `promptless.yaml` and commits the change. When editing manually, only organization admins can edit and save `promptless.yaml` — every member can view it read-only.

As an illustration, here is the shape of a `github_pr` trigger:

```yaml
triggers:
  my-pr-trigger:
    trigger_type: github_pr
    match:
      - repos:
          - acme/docs
        trigger_on:
          - opened           # When PR opens (default)
          - first_approval   # After first approval
          - merge            # When PR merges
        trigger_directories:
          - docs/
```

For the full set of options, see [GitHub PRs](/docs/connect/triggers/github-prs) and the [Configuration Reference](/docs/reference/configuration-reference#triggers).

<Aside type="caution">
  The `github_pr` trigger supports exactly three events — `opened`, `first_approval`, and `merge` — only. There is no "updated"/synchronize/re-review-on-push event today, so you can't configure automatic re-review when new commits are pushed. To re-review after new commits, request it manually via an `@promptless` mention in a PR comment (see [GitHub PRs](/docs/connect/triggers/github-prs)).
</Aside>

<Aside type="note">
  Custom tasks built this way are event-driven: a trigger fires on new events (like a PR opening) and does not retroactively scan or audit your existing pages. For one-time, corpus-wide work such as audits, retrofits, or writing from scratch, use [Pay down docs debt with Deep Analysis](/docs/get-the-most-out/pay-down-docs-debt).
</Aside>

## How a run unfolds

Once the workflow and trigger are in place, a custom task runs the same way every time:

1. The trigger fires on a qualifying event (for a `github_pr` task, a matching pull request).
2. Promptless reads the relevant context for that event.
3. It follows your written workflow step by step.
4. It produces the output your instructions define.

The specifics — what context is read and what output lands — depend on the task; see a worked example for the concrete details.

<Aside type="note">
  Your instructions can describe steps that go beyond what Promptless can currently execute for a given trigger. Promptless does the parts it supports today and treats the rest as intended design. Each worked example's "What's available today" section spells out the current limits for that task, so check it for specifics.
</Aside>
