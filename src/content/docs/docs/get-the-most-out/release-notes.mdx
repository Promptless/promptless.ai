Changelogs and release notes drift the same way the rest of your docs do — they lag behind what actually shipped. Promptless keeps them current alongside everything else: it adds changelog entries when new features ship, updates your existing changelog structure as changes land, and follows the conventions you set for how entries are written. This page shows how to point Promptless at your changelog and set up the triggers that keep it fresh.

<Aside type="note">
This page is about **generating** changelog and release-note content. It doesn't cover *when* a suggestion is drafted or published — for that, see how triggers fire on the changes you choose, and how [publishing policies](/docs/reference/configuration-reference#policies) control PR creation and auto-merge.
</Aside>

## What Promptless does with changelogs

Promptless adds changelog entries automatically when new features ship. If your docs already have a changelog structure, it updates that structure rather than imposing a new one, so entries land in the format your project already uses.

## Point Promptless at your changelog

Tell Promptless where your changelog lives so it can keep that content current:

- **Scope a doc collection to the changelog file.** A collection's `filter` path scope accepts individual files, so an entry like `CHANGELOG.md` lets Promptless update only that file. See [GitHub repos (docs as code)](/docs/connect/doc-locations/github-repos) for path scope.
- **Use a separate collection when your changelog lives elsewhere.** If your changelog sits apart from your main docs, create a separate doc collection for it. See the [open-source quickstart](/docs/start-here/open-source-quickstart) for an example of splitting a changelog into its own collection.

## Trigger changelog updates

Point a trigger at the changes that should refresh your changelog:

- **Watch the changelog directory.** GitHub PR and commit triggers accept `trigger_directories`, so documentation updates fire only when changes touch a changelog directory. This keeps changelog work scoped to the commits that actually affect it.
- **Capture direct commits.** A [GitHub commit trigger](/docs/connect/triggers/github-commits) fires on commits pushed to a monitored branch, so it catches changelog-worthy changes even when they land without a pull request.

## Publish changelog updates immediately

For changelog updates you don't need to review by hand, let them publish on their own. Auto-merge — available for commit triggers, and requiring automatic PR creation — merges a documentation PR into the default branch as soon as it's created. The [github-commits](/docs/connect/triggers/github-commits) page lists "changelog-driven updates" as a use case for this, when you want changelog entries to publish immediately. Configure it in your [publishing policies](/docs/reference/configuration-reference#policies).

## Teach Promptless how you write changelog entries

Promptless follows the conventions you give it for changelog content:

- **Keep changelog entries in their own suggestion.** Feedback like "I always want changelog entries in their own suggestion—please move" is a documented example of the kind of convention you can teach through [feedback](/docs/tune/teaching-conventions).
- **Prefer a skill for changelog entries.** In your Knowledge Base `PROMPTLESS.md`, tell the agent to prefer a particular skill when writing changelog entries. See [How Promptless learns your docs](/docs/connect/doc-locations/how-promptless-learns-your-docs) for how that guidance is applied.
