Changelogs and release notes drift the same way the rest of your docs do — they fall behind what actually shipped. Promptless keeps them current alongside everything else: it adds changelog entries as new features ship, updates your existing changelog structure as changes land, and follows the conventions you set for how entries read. This page shows how to point Promptless at your changelog from the Configuration page and set up the trigger that keeps it fresh.

<Aside type="note">
  This page is about **generating** changelog and release-note content. It doesn't cover *when* a suggestion is drafted or published — for that, see [Release timing](/docs/tune/release-timing) and how [publishing policies](/docs/reference/configuration-reference#policies) control PR creation and auto-merge.
</Aside>

## What Promptless does with changelogs

Promptless adds changelog entries automatically as new features ship. If your docs already have a changelog structure, it updates that structure instead of imposing a new one, so entries land in the format your project already uses.

## Point Promptless at your changelog

Tell Promptless where your changelog lives so it can keep that content current:

- **Scope a doc collection to your changelog.** On the Configuration page, open your doc collection under **Doc collections**, expand **More filters**, and name your changelog path under **Only watch these folders** ("Leave empty to watch the whole repo"). This field also accepts an individual file, so an entry like `CHANGELOG.md` limits Promptless to that one file. See [GitHub repos (docs as code)](/docs/connect/doc-locations/github-repos) for how path scope works.
- **Add a separate collection when your changelog lives elsewhere.** If your changelog sits apart from your main docs, use **Add doc collection** on the Configuration page to track it on its own. See the [open-source quickstart](/docs/start-here/open-source-quickstart) for an example of splitting a changelog into its own collection.

## Watch the changes that update your changelog

Set up a trigger so Promptless activates on the changes that should refresh your changelog:

- **Limit a trigger to your changelog folder.** When you add a **GitHub pull requests** or **GitHub commits** trigger under **Triggers** on the Configuration page, expand **More filters** and use **Only when files in these folders change** to point the trigger at your changelog directory. Promptless then activates only on changes that touch it, so changelog work stays scoped to the commits that affect it.
- **Capture direct commits.** Add a **GitHub commits** trigger ("Run when new commits are pushed to a branch") so Promptless catches changelog-worthy changes even when they land without a pull request. See [GitHub commits](/docs/connect/triggers/github-commits).

## Publish changelog updates immediately

For changelog updates you don't need to review by hand, let them publish on their own. On the **Policies** tab of the Configuration page, turn on **Merge the pull request automatically** ("Skip review and merge Promptless's PRs as soon as they're created"), which also needs **Open a pull request automatically** left on. The [GitHub commits](/docs/connect/triggers/github-commits) page calls out changelog-driven updates as a good fit for this, when you want entries to publish the moment they're ready.

## Teach Promptless how you write changelog entries

Promptless follows the conventions you give it for changelog content:

- **Keep changelog entries in their own suggestion.** Feedback like "I always want changelog entries in their own suggestion—please move" is a documented example of the kind of convention you can teach through [feedback](/docs/tune/teaching-conventions).
- **Point to a skill for changelog entries.** In your **Agent Knowledge Base**, tell the agent to prefer a particular skill when writing changelog entries. See [How Promptless learns your docs](/docs/connect/doc-locations/how-promptless-learns-your-docs) for how that guidance is applied.
