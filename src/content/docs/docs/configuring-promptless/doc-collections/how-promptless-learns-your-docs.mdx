Before creating the first suggestion, Promptless analyzes your existing docs to understand the product, writing style, and how your docs are organized.

### Initial Ingestion

- Builds a searchable index to find relevant content when analyzing triggers
- Maps relationships between pages, sections, and topics

### Product Ontology

Promptless builds an understanding of your product's structure - what features you have, how they relate to each other, and the terminology your team uses. This **product ontology** helps Promptless:

- Know which documentation pages are relevant when a specific feature changes
- Understand your product's concepts and terminology
- Identify connections between features that might require coordinated documentation updates

For example, if your product has a "Workspaces" feature with sub-features like "Permissions" and "Invitations," Promptless understands these relationships and can update all relevant pages when workspace behavior changes.

### Voice and Style Learning

Promptless analyzes how your documentation is written - tone, sentence structure, formatting patterns, and terminology choices. This powers **Voice Match**, ensuring suggestions sound like they were written by your team.

Promptless learns patterns like:

- Whether you write formally or conversationally
- The specific words and phrases you use for concepts
- How you organize content, use headings, and format examples
- Your preferences for capitalization, punctuation, and code formatting

When drafting content, Promptless references similar existing pages to match their style.

### Continuous Learning

Promptless continues learning as you use it. When you edit suggestions or provide feedback, Promptless learns your preferences for future suggestions. As your documentation grows and changes, Promptless adapts to your evolving style and automatically re-indexes when you publish updates to stay current.

### Custom Agent Instructions with AGENTS.md

You can provide persistent guidance to Promptless by creating an `AGENTS.md` file in the root of your docs repository. Use this file to specify project-specific conventions, terminology, and documentation standards that the agent follows when creating suggestions.

#### How It Works

When Promptless starts working on your documentation, it checks the root of your docs repository for an `AGENTS.md` file. If found, the content is included in the agent's instructions, applying your custom guidance to every suggestion.

#### What to Include

Use your `AGENTS.md` file to communicate:

- **Terminology preferences** - Specific words or phrases your team uses for concepts
- **Formatting conventions** - How to structure code examples, callouts, or headings
- **Content guidelines** - What information should always be included (or excluded)
- **Style rules** - Voice, tone, or writing patterns unique to your documentation

#### Example AGENTS.md

```markdown
# Documentation Guidelines

## Terminology
- Use "workspace" not "project" when referring to user containers
- Always capitalize "API" but lowercase "endpoint"
- Prefer "authentication" over "auth" in prose

## Formatting
- Use admonitions for warnings and important notes
- Include a "Before you begin" section in procedural docs
- Code examples should include comments explaining each step

## Style
- Write in second person ("you") rather than third person
- Keep sentences under 25 words when possible
- Use active voice
```

#### Relationship with CLAUDE.md

If your repository already has a `CLAUDE.md` file (used by Claude Code CLI), Promptless uses that instead. The `AGENTS.md` file is only loaded when no `CLAUDE.md` is present, maintaining compatibility with existing Claude Code configurations.

### Custom Skills for Doc Agents

Beyond static instructions in `AGENTS.md`, you can create executable skills that Promptless doc agents automatically discover and invoke. Skills encode repeatable documentation workflows, formatting conventions, operational procedures, or content patterns that the agent runs when a task matches the skill's description.

#### How It Works

Promptless looks for skills in three directories in your docs repository, scanned in this order:

1. `.claude/skills/`
2. `.agents/skills/`
3. `.cursor/skills/`

Each skill is a directory containing a `SKILL.md` file with YAML frontmatter. The filename is case-sensitive and must be exactly `SKILL.md`—a lowercase `skill.md` is not loaded.

```
your-docs-repo/
├── .claude/
│   └── skills/
│       └── api-docs-format/
│           └── SKILL.md
├── .agents/
│   └── skills/
│       └── changelog-format/
│           └── SKILL.md
└── .cursor/
    └── skills/
        └── component-docs/
            └── SKILL.md
```

All three directories use the same Agent Skills format, so a skill works without changes no matter which agent it was authored for. This is convenient if you already keep skills under `.cursor/skills/` for use in Cursor.

When Promptless starts a documentation task, it scans these directories for skills. If a skill's description matches what the agent is about to do, Promptless invokes that skill instead of deriving the workflow from scratch.

When the same skill name appears in more than one directory, the first match in scan order wins: a skill in `.claude/skills/` or `.agents/skills/` takes precedence over a same-named skill in `.cursor/skills/`. This lets you keep a Promptless-specific version of a skill in `.claude/skills/` that overrides a more general one in `.cursor/skills/`.

#### Skill File Format

A `SKILL.md` file has two parts: YAML frontmatter with metadata, and markdown content with the skill's instructions.

```markdown
---
name: api-docs-format
description: |
  Use when writing or updating API reference documentation.
  Ensures consistent formatting for endpoints, parameters, and response examples.
---

# API Documentation Format

When documenting API endpoints, follow these conventions:

## Endpoint Structure
- Start with the HTTP method and path: `GET /api/v1/users`
- Include authentication requirements
- List all parameters in a table

## Response Examples
- Show both success and error responses
- Include realistic example data
```

The `description` field determines when the skill applies. The agent reads all skill descriptions and invokes the one that best matches the current task.

#### Examples

- **Changelog formatting**: Enforce a specific changelog structure, categories, and linking conventions.
- **Component documentation**: Define how to document UI components, including required sections for props, usage examples, and accessibility notes.
- **Code example style**: Specify how code snippets should be formatted, commented, and which languages to use.
- **Translation workflow**: Instructions for preparing content for localization, including which elements to mark as translatable.

#### Skills vs. AGENTS.md

Use `AGENTS.md` for general guidelines that apply to all documentation work. Use skills for specific, repeatable workflows that run only when the task matches. Skills are invoked programmatically; `AGENTS.md` content is always included in the agent's context.
