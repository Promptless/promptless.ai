# Starport template migration — plan

**Goal (Manny, 2026-07-08):** migrate promptless.ai onto the maintained
`Promptless/starport-template` Starlight template so Promptless can perform
**managed migrations** as new template versions ship. Hard constraint from the
opening ask: the migration **must not change the current docs' content or
information architecture**.

This directory is the internal engineering plan for that migration. It is not
built into the site (it lives under `docs/`, the website Agent Knowledge Base,
not `src/content/`). The decision records that back each phase live in
[`adrs/`](adrs/).

## The four decisions (locked in by Manny)

| # | Decision | Choice | ADR |
|---|---|---|---|
| 1 | Adoption model | **Generate-fresh-and-port** — scaffold a fresh site from the template, then port our content and config into it | [ADR 0001](adrs/0001-adopt-starport-template.md) |
| 2 | Scope of "external docs" | **The whole site** — docs + marketing homepage + blog (44 posts) + changelog + pricing + legal + custom components + article/launch-post automation | [ADR 0002](adrs/0002-whole-site-scope.md) |
| 3 | Nav pipeline | **Move to the template's folder-autogenerated topics** — retire the frontmatter-driven `generate:manifest` sidebar pipeline | [ADR 0003](adrs/0003-folder-autogenerated-nav.md) |
| 4 | Capabilities to adopt | **Every optional template capability except i18n** — OpenAPI, sidebar-topics, MCP server (`/mcp` SSR via the Vercel adapter), `starlight-llms-txt`, and build-time link validation; i18n scaffolding removed | [ADR 0004](adrs/0004-adopt-capabilities.md) |

## The tension we are holding, and how we resolve it

Decisions 1 and 3 pull against the hard constraint. The site must look and
resolve **exactly** as it does today — same pages, same nav labels and order,
same URLs — even as its nav mechanic changes from our `generate:manifest`
pipeline to the template's folder-autogenerated `starlight-sidebar-topics`.

The migration is only mergeable if it preserves the IA with **zero
user-visible diff**. [ADR 0003](adrs/0003-folder-autogenerated-nav.md) spells
out the full reconciliation; the short version is that our current URLs and nav
placement already derive from the same signals Starlight's native
autogeneration reads:

- **URLs are preserved for free.** Docs already live at
  `src/content/docs/docs/<section>/<page>.mdx` with `slug: docs/<section>/<page>`
  mirroring the file path. Starlight's `autogenerate: { directory: 'docs' }`
  produces the same `/docs/...` routes from the same tree, so no slug changes —
  and no redirects — are needed for the nav-mechanic switch itself.
- **Order and per-page labels are preserved for free.** Placement, visibility,
  and label already come from each page's `sidebar.order`, `sidebar.hidden`,
  and `sidebar.label` frontmatter. Starlight's built-in autogeneration reads
  those same three fields natively; our custom `generate:manifest` reads
  nothing the platform does not already read.
- **The one real mechanic shift:** section/group **labels** move from the
  section index page's frontmatter into explicit `label`s in the
  `starlight-sidebar-topics` config in `astro.config.mjs`. Starlight's raw
  autogeneration title-cases the folder name, turning *"Get the most out"* into
  *"Get The Most Out"* and *"Work the queue"* into *"Work The Queue"*. Setting
  each group's `label` explicitly in config reproduces the exact current labels.
  This is the item to watch on the nav-switch phase.

If any nav placement genuinely cannot be reproduced without a visible change,
that is a stop-and-escalate — we surface it rather than change the IA silently.

## Engine / dependency gap

| | Current (`promptless.ai`) | Template (`starport-template`) |
|---|---|---|
| Astro | `^5.13.2` | `^7.0.5` |
| Starlight | `^0.37.6` | `^0.41.2` |
| Node | `>=20.0` | `>=22.12.0` |
| Nav | custom `generate:manifest` | `starlight-sidebar-topics` + folder autogenerate |
| MCP | route-based `llms.txt` / `.md` endpoints | vendored `@starport/starlight-mcp` SSR plugin at `/mcp` |
| OpenAPI | `starlight-openapi@^0.22` | `starlight-openapi@^0.26` |
| `llms.txt` | hand-rolled route endpoints (`src/pages/llms.txt.ts`, `.md`) | `starlight-llms-txt` plugin (overlaps our endpoints — reconcile) |
| Link validation | `check-broken-links` skill (`linkchecker`, non-blocking) | `starlight-links-validator` (build-failing) + our skill |
| SSR adapter | none (fully static) | `@astrojs/vercel@^11` (needed for on-demand `/mcp`; Vercel, not Cloudflare) |
| i18n | none | template ships `defaultLocale`/`locales` + sample `es/` tree (**not adopted** — removed on port) |

The engine jump (Astro 5 → 7, Starlight 0.37 → 0.41, Node 20 → 22) is a
breaking upgrade that touches every custom component and the 7 Starlight
overrides. It is the highest-risk part of the migration and is phased first,
in place, before any template scaffolding lands.

## How the work is split

Phased, per-increment migration — the same shape as the IA cut-over
(`docs/content_strategy/information_architecture/implementation-plan.md`), which
this repo has already run successfully. Each phase is one or a few PRs a
reviewer can evaluate in isolation, and **each phase must produce zero
content/IA/URL diff before it merges** (verified per §Definition of done). No
"v2 site" tree ships in parallel; we migrate the one live site in place.

| Phase | Increment | What lands | Verified by |
|---|---|---|---|
| **0** | This plan + ADRs | The four ADRs and this plan. No code changes. | Review only |
| **1** | Engine upgrade in place | Bump Astro 5 → 7, Starlight 0.37 → 0.41, Node 20 → 22; fix breaking changes across the 54 components and 7 Starlight overrides. Nav mechanic **unchanged** this phase. | `npm run check`, smoke tests, broken-link check, visual diff |
| **2** | Template scaffolding | Add `.starport-template.json` (the version marker Promptless reads for managed migrations, `templateVersion: 1.0.0`); adopt the template's ADR/`CUSTOMIZE` conventions. Do **not** bring in the template's i18n scaffolding (`defaultLocale`/`locales` + sample `es/` tree). No user-visible change. | Marker present; no i18n config; build clean |
| **3** | Nav mechanic switch | Replace `generate:manifest`'s `sidebar.json` with `starlight-sidebar-topics` + `autogenerate`, with explicit group `label`s reproducing current labels/order. Retire `buildSidebar()`; keep `route-manifest.json` generation (the `.md` endpoints still need it). | Byte-level nav + URL diff = 0 |
| **4** | OpenAPI on the template | Move the API reference to `starlight-openapi@0.26` under the template's topic wiring; keep `/api/*` URLs identical. | `/api/*` unchanged |
| **5** | Sidebar-topics | **Settled: single "Documentation" topic, no topic-switcher bar.** The nav mechanism (`starlight-sidebar-topics` hosting one topic) landed in Phase 3 and the OpenAPI groups moved into it in Phase 4; today's site surfaces one docs tree and the repo's `Sidebar` override never mounts the plugin's switcher. Phase 5 formalizes that a top-level topic bar and a separate "API Reference" topic are **not** introduced (both would be a user-visible IA change), and removes the "deferred to Phase 5" markers in `astro.config.mjs` and the ADRs. No config or rendering change. | Nav diff = 0 (260 == 260 routes) |
| **6** | MCP server + `llms.txt` plugin | Wire the vendored `@starport/starlight-mcp` SSR plugin + `@astrojs/vercel` adapter (Vercel, not Cloudflare) to serve `/mcp`; adopt `starlight-llms-txt`; reconcile both against the existing hand-rolled `llms.txt` / `llms-full.txt` / `.md` endpoints (keep or supersede per route, decided in the phase). | `/mcp` responds; `llms.txt`/`.md` URLs + content unchanged; static routes unchanged |
| **6b** | Build-time link validation | Enable `starlight-links-validator` after the engine upgrade has stabilized; remediate or configure around any pre-existing links it rejects (hidden pages, redirect-only links); reconcile scope with `check-broken-links`. | Build passes with validator on; no new 404s |
| **7** | Cut-over completion | Remove dead scaffolding, confirm `.starport-template.json` is the sole source of truth for template version, update `AGENTS.md` Documentation Map. | Full crawl; final review |

Phases 1–2 are prerequisites for everything after. Phases 3–6 are loosely
coupled and can be reviewed largely independently once the engine and marker
are in place. **The non-docs surface (homepage, blog, changelog, pricing,
legal, automation) has no home in the docs-only template and is carried as-is
through every phase** — see [ADR 0002](adrs/0002-whole-site-scope.md).

Two of the newly-confirmed capabilities carry reconciliation work against
surface we already own, and are the watch items for this capability set
([ADR 0004](adrs/0004-adopt-capabilities.md)):

- **`starlight-llms-txt` collides with our hand-rolled endpoints.** The plugin
  generates `/llms.txt` and `/llms-full.txt` — the exact routes we already serve
  from `src/pages/llms.txt.ts` / `llms-full.txt.ts` (plus per-page `.md`). Phase 6
  picks one producer per route and confirms the emitted URLs and content stay
  identical; our endpoints also cover the non-docs Markdown variants
  (`index.md.ts`, `pricing.md.ts`, `free-tools.md.ts`), which any superseding must
  not drop.
- **`starlight-links-validator` is a build-failing gate.** It is stricter than the
  non-blocking `check-broken-links` skill and may reject links that pass today
  (links into `sidebar.hidden` pages, redirect-only links). Enable it after the
  engine upgrade has stabilized (Phase 6b), and reconcile its scope with the
  existing skill.

## Definition of done (every phase)

A phase is mergeable only when, for the pages it touches:

- `npm run build` (or `npm run check`) builds the nav, route manifest, and
  content collections clean.
- The **check-broken-links** workflow against a local `npm run dev` shows no
  new 404s and every historical URL still resolves (via route or redirect).
- `npm run test:smoke` stays green.
- The rendered nav (labels, order, grouping) and the published URL set are
  **byte-for-byte identical** to `main` for docs pages — the migration changes
  the mechanic, not the output.

## What this run delivered vs. what needs engineering execution

This is a **planning increment (Phase 0)**. It ships the decision records and
the phased plan. It does **not** perform the Astro 5 → 7 engine upgrade, port
the 54 components, switch the live nav mechanic, or wire the SSR MCP server —
each of those is a code migration that must be built and verified against a
running build (Phases 1–6), not authored blind. Those phases are scoped here so
they can be executed and reviewed as independent PRs.

## Knowledge-base guidance that this migration invalidates

The shared `PROMPTLESS.md` and this repo's `AGENTS.md` both document the current
nav convention: `sidebar.json` is frontmatter-generated via `generate:manifest`
and must not be hand-edited. **Phase 3 intentionally retires that pipeline.**
When Phase 3 lands, that guidance must be updated to describe the folder-
autogenerated `starlight-sidebar-topics` mechanic instead. Flagged to dispatch;
the LTC / `PROMPTLESS.md` edit is dispatch-owned and should follow the nav
switch, not precede it.
