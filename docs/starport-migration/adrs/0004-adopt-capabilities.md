# ADR 0004 — Adopt every optional template capability except i18n

- Status: accepted
- Date: 2026-07-08
- Deciders: Manny
- Context source: #docs Slack thread, 2026-07-08 (initial ask + "Yes" capability confirmation)

## Context

The template ships several optional capabilities: an OpenAPI API reference
(`starlight-openapi`), multi-section nav (`starlight-sidebar-topics`), a read-only
`/mcp` SSR server (`@starport/starlight-mcp`), an agent-friendly `llms.txt`
generator (`starlight-llms-txt`), build-time link validation
(`starlight-links-validator`), and i18n / localization.

The first cut of this ADR recorded adopting only three of them (OpenAPI,
sidebar-topics, MCP). Manny then confirmed the full optional-capability set with
an explicit "Yes": **enable every optional capability the template offers except
i18n.** This ADR is updated to that decision.

## Decision

**Adopt all of the template's optional capabilities except i18n:**

1. OpenAPI API reference (`starlight-openapi`) — enabled.
2. Multi-section nav / sidebar-topics (`starlight-sidebar-topics`) — enabled.
3. Docs MCP server (`@starport/starlight-mcp`, read-only `/mcp` route with
   `search` + `get_page`) — intended, served via the **Vercel** SSR adapter
   (`@astrojs/vercel`), the template's default and verified MCP target.
   **Deferred out of Phase 6** — see the update note below.
4. Agent-friendly `llms.txt` (`starlight-llms-txt`) — enabled (the template
   treats it as core).
5. Build-time link validation (`starlight-links-validator`) — enabled.

**Do not adopt i18n.** Remove the `defaultLocale` / `locales` config and the
sample `es/` locale content tree the template ships. Manny does not need
localization now.

### Update (2026-07-10): MCP deferred, `llms.txt` plugin adopted

Manny made two calls that revise how capabilities 3 and 4 land in Phase 6:

- **`llms.txt` plugin over the hand-rolled endpoints.** "Use the plugin, not my
  hand-rolled llms.txt. The whole point of this initiative is to get away from
  hand-rolling things." Phase 6 adopts `starlight-llms-txt` and retires the
  hand-rolled `src/pages/llms.txt.ts` / `llms-full.txt.ts` route endpoints,
  reconciling their non-docs coverage on the plugin's terms (see §4).
- **MCP server deferred.** "Attempt phase 6 again, but drop the MCP server
  capabilities. We'll attempt to add them again later." The
  `@starport/starlight-mcp` plugin, the `/mcp` route, and the `@astrojs/vercel`
  SSR adapter are **not** adopted in Phase 6; the site stays fully static.
  Capability 3 remains an accepted target for a later attempt, not a rejection.

## The capabilities

### 1. OpenAPI support

We already render an OpenAPI reference via `starlight-openapi@0.22` (the API
trigger reference at `/api/*`, `schema: ./public/openapi/api-triggers.yaml`).
Adopting the template means moving to `starlight-openapi@0.26` and wiring the
generated `openAPISidebarGroups` through `starlight-sidebar-topics` instead of
spreading them into a flat `sidebar` array.

- **URL guardrail:** `/api/*` routes must stay identical. The `base: 'api'`
  config is preserved; only the plugin version and the sidebar-wiring change.
- Note: `Promptless/PROMPTLESS.md` records a deliberate decision **not** to
  document the `doc_collection_id` field in the public API reference. This
  migration is a rendering/engine change and must not alter the API spec's
  documented surface.

### 2. sidebar-topics

`starlight-sidebar-topics` provides top-level topic switchers, each with its own
sidebar. In the template it wraps the folder `autogenerate` groups. For us it is
primarily the mechanism that hosts the folder-autogenerated docs nav
([ADR 0003](0003-folder-autogenerated-nav.md)) and, in Phase 4, the OpenAPI
sidebar groups.

- **IA guardrail:** the site today has a single docs sidebar, not a visible
  multi-topic switcher bar. Introducing topics must not add a user-visible topic
  bar the current site lacks unless it renders identically. Default to a single
  "Documentation" topic that reproduces today's nav exactly; add an "API
  Reference" topic only if it matches how `/api/*` is surfaced today.
- **Phase 5 resolution (settled):** a single "Documentation" topic, with **no
  topic-switcher bar and no separate "API Reference" topic**. `/api/*` is
  surfaced inside the one docs tree today, not as a distinct topic, so promoting
  it to its own topic — or exposing the switcher bar at all — would be a
  user-visible IA change the guardrail forbids. The mechanism landed in Phase 3
  (the topic hosts the folder-autogenerated nav) and Phase 4 (the OpenAPI groups
  moved into it); the repo's `Sidebar` override renders the sublist directly and
  never mounts the plugin's `<Topics/>` switcher, so the rendered nav stays
  byte-for-byte identical to the pre-topics site. Phase 5 is a decision phase: it
  records this outcome and removes the "deferred to Phase 5" markers, with no
  config or rendering change.

### 3. MCP server (`/mcp` SSR, Vercel adapter) — deferred out of Phase 6

> **Deferred (2026-07-10).** Manny dropped the MCP server from Phase 6 ("drop
> the MCP server capabilities. We'll attempt to add them again later"). The
> `/mcp` route, the `@starport/starlight-mcp` plugin, and the `@astrojs/vercel`
> SSR adapter are not adopted in this phase, so the deploy shape stays fully
> static. The design below is retained as the target for a later attempt.

The template vendors `@starport/starlight-mcp`, a Starlight plugin that serves
the docs as a read-only MCP server over Streamable HTTP at `/mcp`. It exposes
`search` and `get_page` tools plus an `llms.txt`-style index, builds its content
index at build time into a Vite virtual module, and injects the `/mcp` route
with `prerender: false`.

- **Vercel adapter, not Cloudflare.** `/mcp` is on-demand, so the site needs an
  SSR adapter. We use `@astrojs/vercel` — the template's default `MCP_RUNTIME`
  and its verified target. The template also documents a Cloudflare option, but
  it is experimental/unverified there, so we do not use it. promptless.ai already
  deploys on Vercel (and is fully static today with no adapter), so `@astrojs/vercel`
  is both the low-risk and the verified choice; adding the adapter is a
  deploy-shape change to validate on Vercel (plan §Phase 6).
- **Reconcile with what we already have, now including `starlight-llms-txt`.** We
  already serve `llms.txt`, `llms-full.txt`, and per-page `.md` via route
  endpoints (`src/pages/*.ts`), and we are now also adopting the
  `starlight-llms-txt` plugin (below). Phase 6 decides whether `/mcp` and the
  plugin supersede or complement the hand-rolled endpoints; do not remove the
  existing endpoints without confirming nothing depends on them and the emitted
  URLs/content stay identical.
- v1 of the MCP server is authless (all pages public). That matches our fully
  public docs, so no gating work is needed now.

### 4. Agent-friendly `llms.txt` (`starlight-llms-txt`)

The template treats `starlight-llms-txt` as core: it generates `/llms.txt` and
`/llms-full.txt` from the docs content at build time.

- **Direct route overlap with our hand-rolled endpoints — resolved: the plugin
  supersedes.** We served `/llms.txt` and `/llms-full.txt` from
  `src/pages/llms.txt.ts` and `llms-full.txt.ts`. Enabling the plugin put two
  producers on those routes. Phase 6 resolves this by adopting the plugin and
  **retiring both hand-rolled `.txt` endpoints** (Manny, 2026-07-10). Both routes
  keep resolving, now plugin-generated; the plugin also emits a new
  `/llms-small.txt` (an abridged set), which is the only route-set addition.
- **The per-page `.md` endpoints are a separate surface and are kept.** The
  plugin does not generate per-page Markdown or the marketing/pricing Markdown
  variants, so `[...slug].md.ts`, `index.md.ts`, `pricing.md.ts`, and
  `free-tools.md.ts` remain untouched — their URLs and content are unchanged.
- **Non-docs coverage reconciled on the plugin's terms.** The old hand-rolled
  index also listed the marketing homepage, blog, and changelog, which the
  docs-scoped plugin does not crawl. That coverage is preserved by configuring
  the plugin's `optionalLinks` (Blog, Changelog, and the marketing `.md`
  variants) rather than by keeping a hand-rolled route.

### 5. Build-time link validation (`starlight-links-validator`)

`starlight-links-validator` runs at build time and **fails the build** on broken
internal links and, optionally, invalid anchor fragments.

- **New failing gate on top of `check-broken-links`.** The repo already has the
  `check-broken-links` skill (a `linkchecker` crawl of a local dev build). The
  plugin is a stricter, build-blocking check. It may flag links that pass today —
  e.g. links into `sidebar.hidden` pages, or links that only resolve through the
  hand-maintained `redirects.json` / static redirect map rather than a live
  route. Enable it **after** the engine upgrade stabilizes (not during Phase 1),
  and budget for remediating or configuring around any pre-existing links it
  rejects. Reconcile its scope with `check-broken-links` so we are not
  maintaining two overlapping link checks with different verdicts.

## Not adopted

- **i18n / localization.** No current localization need. Remove the template's
  `defaultLocale` / `locales` config and its sample `es/` locale content tree
  during the port. (promptless.ai has no i18n config or locale tree today, so
  this is a "do not bring the template's scaffolding in" step, not a removal from
  our own site.) Revisit only if the localization CUJ is scheduled.

## Consequences

- New runtime dependency surface: `starlight-sidebar-topics`, upgraded
  `starlight-openapi`, `starlight-llms-txt`, and `starlight-links-validator`.
  The SSR adapter (`@astrojs/vercel@^11`) is **not** added in Phase 6 — it comes
  in only when the deferred MCP server is attempted.
- Deploy shape **stays fully static** in Phase 6. The MCP server was the only
  on-demand route, and it is deferred; without it no SSR adapter is needed, so
  the static build shape is unchanged.
- One capability created reconciliation work that is now resolved:
  `starlight-llms-txt` overlapped our hand-rolled `llms.txt` endpoints; Phase 6
  adopts the plugin and retires those two `.txt` endpoints (keeping the per-page
  `.md` endpoints). `starlight-links-validator` still adds a build-failing link
  gate on top of `check-broken-links`, sequenced after the engine upgrade
  (Phase 6b).
- Phases 4–6 each carry a URL/nav-preservation acceptance test; Phase 6's is
  that `/llms.txt` and `/llms-full.txt` keep resolving (now plugin-generated),
  the only route-set change is the added `/llms-small.txt`, and every HTML page
  and per-page `.md` URL stays identical.
