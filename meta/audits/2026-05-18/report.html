Executive summary

The 2026-05-18 audit evaluated promptless.ai's product documentation — 59 MDX pages across 9 sections — through a 12-lens framework covering content, structure, technical health, accessibility, agent-readability, instrumentation, drift, theater/AI-tells, framework fit, audience coverage, and journey/task success.

The audit produced 37 active findings: 2 Critical (a truncated helm upgrade command and a Trust & Safety gap on per-call model transfer), 11 Significant (lifecycle gaps: first-success orchestration, decision-support, maintain/update; plus a site-wide WCAG color-contrast failure and a near-total absence of docs-page instrumentation), 19 Moderate, and 5 Low. Three findings were withdrawn as the audit's discipline caught two false positives and one out-of-scope drift.

The audit ran in five evolution passes — degraded shell-only → Node-enabled re-validation → expansion with Vale/AFDocs/Seven Action Model → synthesis (canonical page-data store; DARI spec; new execution profile) → Phase 8 fold into Phase 13. The pass structure is itself the audit's most valuable methodological output: the next audit on this set should collapse to a single execution after preflight establishes capabilities.

Beyond findings, the audit shipped a reusable checklist, a tooling-spec sequenced across 15 build items, a canonical per-page data store (evidence/page-data.csv + page-data.json) that joins every lens's per-page output, and the DARI-docs agent-simulation spec'd for the next audit cycle. Phase 8 (Procedural testing scope) was folded into Phase 13 after a reviewer correction: Doc Detective pass/fail is task-success evidence, not a counting exercise.

Top priorities (T1 in the tooling-spec sequencing): fix the truncated helm command (F-0001); fill the model-transfer specifics (F-0002); fix the footer contrast (F-0030); add the 6 missing doc descriptions and 2 sidebar orphans (F-0003 / F-0004); ship the 8 docs analytics events (F-0008). These five items unblock evaluation, security review, accessibility, agent comprehension, and quarterly improvement.

Process narrative

The audit ran in five evolution passes. The first three were execution; the last two were synthesis + structural refactor.

Pass 1 — Degraded mode (shell-only)

Node.js was unavailable in the audit environment. The audit ran with shell tools only: ripgrep, git, python3 (stdlib only — no PyYAML), jq, awk, sed. AST-dependent lenses (MDX parsing, lychee link checking, pa11y, Vale, markdownlint) were blocked. Phase 1's inventory ran via a hand-written Python script that parsed YAML frontmatter and used targeted regex for known MDX shapes (<Frame>, <Steps>, image tags). 29 findings landed in this pass, with degraded-evidence tags marking every finding that depended on the unavailable tooling.

Pass 2 — Node-enabled re-validation

Node 24.15.0 was installed mid-audit. A full AST inventory ran with gray-matter + remark-mdx. Two findings were corrected: F-0005 (alt-text gaps) was withdrawn — the regex parser was looking for alt= on <Frame> tags but the alt text lived on the inner <img> child; AST confirmed 100% alt coverage. F-0018 (untagged code fences) dropped from 45 to 8; the regex had counted closing fences and indented fences as opens. npm run check and the smoke tests passed; pa11y caught a footer color-contrast failure (sampled). The audit's "no regex over MDX" rule and the evidence: degraded-parser tag discipline worked exactly as designed — both false positives were caught and resolved when AST tools became available.

Pass 3 — Expansion with full toolchain

The user installed gh CLI + Vale via brew, and provided a canonical reference for the Seven Action Model. The expansion ran the remaining lenses: Phase 2 (IA), Phase 4 (Vale + cspell + alex), Phase 5 (pa11y-ci full sweep over 51 docs URLs), Phase 6 (AFDocs full scorecard via gh), Phase 10 (theater + ownership + AI-tells), Phase 11 (multi-framework classification across all three axes), Phase 12 (strategy-level persona matrix). The full sweep confirmed F-0030 site-wide (153 of 154 pa11y issues are the same footer-contrast pattern on every doc page), found 3 high-signal AI-tell pages (data-handling, deep-analysis, SSO-setup), produced a coverage matrix with 5 strategy-level audience gaps, and surfaced two AFDocs warnings that turned out to be out-of-scope (F-0031 / F-0032 withdrawn).

Pass 4 — Synthesis

Built evidence/build-page-data.mjs — a Node script that merges every per-lens raw output (inventory, instrumentation, classification, ownership, AI-tells, Vale, cspell, pa11y, AFDocs, findings) into a single canonical evidence/page-data.csv (60 lines × ~45 columns) and page-data.json (nested mirror). This is the single front door for "what do we know about page X?". Spec'd dari-docs (Agent task-success simulation) as a Phase 13 sub-lens for the next audit cycle (default: self-managed mode against the user's own dari.dev org). Rewrote the Execution Profile from Core-vs-Expanded staging to run-everything-runnable (skip only for infeasibility or explicit opt-out).

Pass 5 — Phase 8 fold + HTML report (this artifact)

Reviewer correction: "The Doc Detective testing should also fall under Journey & Task Success. Those aren't just about counting Doc Detective tests (especially because tests can be written inline as body content in the docs), but passing/failing tests indicate procedural accuracy." Phase 8 was folded into Phase 13 as the procedural-accuracy sub-lens alongside DARI. Lens count: 13 → 12. The page-data merger gained Doc Detective discovery columns (spec count, inline-marker count, <Steps> block count, test-covered boolean). Discovery results for this audit: 1 spec file (Clerk auth, not a documented procedure), 0 inline markers in any of the 59 MDX files, 17 documented procedures across 10 pages — 0% procedural-accuracy coverage. This HTML report is the synthesis view of all four prior passes plus the fifth-pass refactor.

What we learned about audit infrastructure

  • Containerize the audit (tooling-spec S10). Without Node, 9 of 12 lenses were degraded; with Node, the audit ran in hours not days. A Dockerfile with Node + lychee + pa11y + Vale + Doc Detective + ripgrep + shellcheck pinned would collapse all five passes into one.
  • "No regex over MDX" discipline works. The two false-positive withdrawals (F-0005 alt-text, F-0018 fence-count) were both caught because the original findings carried evidence: degraded-parser tags that flagged them for AST re-verification.
  • Calibration regimes hold under pressure. Phase 10 produced zero "this page is theater" verdicts (only risk indicators with mandatory false-positive notes). Phase 12 produced no per-page persona-mismatch findings without validation. Phase 11's Seven Action axis ran only after the canonical reference was supplied. Phase 13's DARI sub-lens is spec'd-only this run because credentials weren't configured.
  • The page-data merger is the highest-ROI synthesis output. Reviewers asking "what do we know about page X?" had to cross-reference ~9 raw files. Now page-data.csv is the single front door; per-lens raw outputs stay as evidence.

  • Did the global announcement banner get dismissed from a docs URL?
  • Did the mobile menu footer get used while on a docs page?

  • Are users telling us a page was helpful or not?

Specific recommendations for closing the gap

Each is a candidate for Phase 14's tooling-spec.md. Prioritized by ROI:

  1. Auto-instrument page views with structured properties. PostHog autocapture already fires on docs pages, but events typically don't carry the page's frontmatter (slug, title, sidebar section). Add a global layout-level event that fires on docs-page render with { page_type: &quot;docs&quot;, slug, section, hidden, has_description } — that single event unlocks per-section, per-page-type analytics for free.
  1. Add a feedback widget (docs_helpful_voted). A simple "Was this helpful?" with thumbs up/down at the bottom of every page. Stores answer + slug. Three weeks of data tells you which pages are bleeding users. Use Posthog Feedback Widget or a custom 30-line component.
  1. Track code-copy events (docs_code_copied). Wrap fenced code blocks with a copy button that emits an event with { slug, language, block_index }. Tells you which examples are actually being used. Particularly valuable for the 17 &lt;Steps&gt; procedures.
  1. Track search use (docs_searched, docs_search_zero_result). Starlight uses Pagefind by default — capture the query + result-count. Zero-result queries are the highest-signal feedback for content gaps.
  1. Track outbound link clicks (docs_external_link_clicked). Especially for the 28 external links. The pattern is text — wrap markdown links with a tracking handler in the layout, or use a single delegated event listener.
  1. Track integration CTA clicks (docs_integration_connect_clicked). The integration pages all link to app.gopromptless.ai/projects / /integrations. Tracking which integration page generated each click reveals which integrations users want to configure, vs. which they read about and don't.
  1. Log /llms.txt and /llms-full.txt access. Both are dynamic Astro routes (src/pages/llms.txt.ts, src/pages/llms-full.txt.ts) — add a server-side log or analytics call with { ua, url }. Filter by known agent UAs (GPTBot, ClaudeBot, Bingbot, etc.) to measure agent consumption — relevant for an AI documentation product whose customers' agents may be reading the docs.
  1. Per-step completion tracking for &lt;Steps&gt; blocks. Aspirational; lower ROI than the above. Wrap &lt;Steps&gt; to emit a step_visible event when a user scrolls past each step.

Appendix

Severity rubric

Critical
User task failure, data loss, security/privacy exposure, regulatory miss, broken auth/install on a path most users follow. For AI products handling customer code, private data, regulated data, or external model/provider transfer: trust-and-safety gaps qualify.
Significant
Blocks a common task workflow with workaround available; missing topic that a primary persona needs; high-confidence drift on a frequently-touched page; strategy-level audience gap; no instrumentation on a high-value page when there's a stated measurement goal or known user pain.
Moderate
Reduces clarity, slows the user, accumulates docs debt; style/consistency violations; ownership signals; content-type mismatches; readability/translation friction.
Low
Cosmetic, single-page formatting, lint-class issues, individual typos, minor naming inconsistencies that don't impede comprehension.

Finding schema

Every finding follows this structure. Findings without complete fields are rejected during synthesis.

F-NNNN — <short title>
  Lens: one of the 12 lenses
  Severity: Critical | Significant | Moderate | Low
  Page/entity: file path or system entity; "(global)" if cross-cutting
  Evidence: observed signal — quote, command output, file:line, artifact path
  Source command: reference to evidence/raw/... if generated
  Why it matters: user/business impact, 1–2 sentences
  Confidence: High | Medium | Low (+ 1-line rationale)
  Remediation type: Fix | Improve | Consolidate | Remove | Create | Spec-only
  Automation candidate: Yes (tool: ...) | Partial | No (human judgment required)
  False-positive notes: how this could be wrong (required for Low-confidence
                        or judgment-driven findings)
